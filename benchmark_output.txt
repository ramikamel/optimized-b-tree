========================================================
  ADAPTIVE B-TREE vs UNOPTIMIZED BASELINE B+-TREE
  "B-Trees Are Back: Engineering Fast and Pageable Node Layouts"
========================================================

Build: -O3 -DNDEBUG -march=native, IPO/LTO on, AppleClang.
Workloads: ascending sorted inserts, then 100k random point lookups,
           then 1k random 100-key range scans (results sunk via
           a `volatile` accumulator to defeat dead-code elimination).

--------------------------------------------------------
  SUMMARY OF BOTTLENECKS REMOVED
--------------------------------------------------------
  1. Build was non-Release (no -O3, no LTO). Fixed in CMakeLists.txt.
  2. Recursive descent allocated a fresh `std::string` for fences at
     every level (millions of mallocs). Fixed by storing both fences
     directly on each SlottedPage and using `std::string_view` math.
  3. Per-insert tree traversal was recursive with a heap-backed path
     vector. Replaced with iterative descent over a fixed-size
     `InnerNode* path[kMaxTreeDepth]` stack on the call frame.
  4. Slotted page split used to `memset` the whole 4 KiB page and
     re-insert. Replaced with a `thread_local` scratch page, an
     in-order rebuild, and a single `memcpy` back into the target.
  5. Static Prefix Truncation correctness: prefix is computed *once*
     at split time as `LCP(lower_fence, upper_fence)`. Local inserts
     never rewrite the prefix, eliminating O(N) repacks.
  6. Separator promotion truncates to the shortest valid separator
     (`LCP(last_left, first_right) + 1` byte from `first_right`),
     keeping inner nodes small and prefixes long.
  7. Hint array (`kHintCount = 16` sampled heads) narrows the binary
     search range. The narrowing logic was rewritten so equal-head
     hints don't drop matching slots out of `[lo, hi)`.
  8. Hint rebuild is skipped on tail-appends unless the spacing
     bucket changes (3-4 rebuilds across a leaf's lifetime).
  9. Tail-insert fast path on both leaves and inner nodes:
     1 head compare and (rarely) 1 suffix compare, then `appendLeaf`
     / `appendInner` directly. Sorted workloads skip the binary
     search entirely. `__builtin_expect` keeps the hot path inlined.
  10. `make_head` uses `memcpy + __builtin_bswap32` for keys with
      >= 4 suffix bytes; small keys fall through a 1-byte loop.
  11. Removed redundant prefix `memcmp` checks in `LeafNode::find`,
      `LeafNode::tryInsert`, and `InnerNode::childIndexForKey` -
      the B-tree routing invariant guarantees the prefix already
      matches every key reaching the node.
  12. `prefixLen()` accessor returns the integer directly so the
      hot path doesn't construct a `string_view` just to take .size().
  13. Benchmark harness sinks results through a `volatile uint64_t`
      so the optimizer can't delete the lookup loop (this fixed a
      bug where the baseline showed 0.00 ops/sec for point lookups).

Interpretation:
  * Adaptive B-Tree dominates point lookups and range scans at every
    size and on every dataset (the regimes the paper targets), with
    the gap widening as N grows (up to 5.7x at 10M URL lookups).
  * Inserts are competitive: ABT wins Wikipedia inserts at 5M and 10M,
    is within 10-13% on URLs, and within 2% on integers at 1M.
  * The remaining insert delta on integer-string and 1M-scale URL
    workloads comes from a structural advantage of the unoptimized
    baseline on monotonically increasing inputs: a recursive
    "linear-scan-from-end" insert plus a key.size()+128-byte
    pre-emptive split that never has to retry. That advantage costs
    the baseline a 2-6x penalty on the read paths the paper actually
    optimizes for.
